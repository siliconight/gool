# `GoolMultiplayerBridge` — networking integration

Added in v0.46.0. Autoload that wires Godot's `MultiplayerAPI` (or
any custom transport you plug in) to gool's existing replication
and VOIP primitives. Lives at
`res://addons/gool/multiplayer_bridge.gd`, registered as the
`GoolMultiplayerBridge` autoload by the gool plugin.

## TL;DR for the backend lead

You're picking the transport. The bridge doesn't lock anything in.

- **Default behavior**: routes everything through Godot's
  `MultiplayerAPI`. Works out of the box with `ENetMultiplayerPeer`,
  `SteamMultiplayerPeer`, `WebRTCMultiplayerPeer`, or any other
  `MultiplayerPeer` subclass.
- **Custom transport**: set
  `GoolMultiplayerBridge.transport_mode = CUSTOM` and the bridge
  emits `send_*` signals instead of RPCs. Your transport code
  listens, routes via whatever protocol you've built (raw UDP,
  Steam P2P channels, a relay you control), and calls the
  bridge's `receive_*` methods when packets arrive.
- **Mixed mode**: nothing stops you from using `MultiplayerAPI`
  for events and a custom channel for voice — just set
  `route_voice_packets = false` and dispatch voice via your own
  path (e.g. `Gool.submit_voice_packet` directly from Steam
  callbacks).

## How each architecture decision is honored

| Decision (from the network arch doc) | Where the bridge honors it |
|---|---|
| **0ms gunshot latency on firing client** | `fire_predicted_event` calls `Gool.submit_event_local` FIRST, then dispatches replication. Local audio never waits on the network. |
| **Listen-server model** | The listen-server calls `advance_tick(simulation_tick, server_time_ms)` once per tick. Clients call the same locally with the host's last-known tick. |
| **Client prediction with `prediction_id`** | `fire_predicted_event` returns a `prediction_id` generated by `Gool.make_prediction_id`. Game code caches it. On invalidation (server says "missed"), call `Gool.cancel_predicted_event(prediction_id)`. |
| **Sub-tick stamp system (in flight)** | Current API uses `int simulation_tick`. When the C++ side gains sub-tick precision, the bridge will accept an additional `sub_tick_offset_ms`. Tracking issue: v0.47.0. |
| **20–30 Hz tick rate** | `advance_tick` is the only tick-rate signal the bridge depends on. Whatever rate the listen-server runs at is what gool uses. Bridge doesn't care if it's 20, 30, or anything else. |
| **Drop stale events** | `staleness_threshold_ms` (default 250 — matches the existing `play_networked`). Tune up for relayed peers, down for competitive modes. |
| **Two transports day-1 (Steam P2P + ENet)** | `transport_mode` enum has explicit modes; `auto_register_voice_sources` and `route_voice_packets` are independently toggleable so a Steam build can disable bridge voice routing without losing event replication. |
| **Host migration (~5s, 5–10s state loss)** | `capture_network_snapshot()` produces a `GoolNetworkSnapshot` Resource — voice player IDs + mix snapshot + current tick. New host calls `apply_network_snapshot(snap)` after migration. Tier 1; active-emitter rehydration is v0.47.0. |
| **Relays** | The bridge doesn't see them — they're below the transport. But `Gool.get_voice_jitter_ms(peer_id)` lets your game UI surface "this player is relayed" by jitter / loss thresholds. |
| **Anti-cheat is non-goal** | Bridge does ZERO validation. Trusts every replicated event and dispatches to gool. Per the arch doc: "if they cheat, they cheat." |
| **Voice chat is Release 3** | Voice routing in the bridge is implemented but `route_voice_packets` defaults to `true`. Game code that doesn't need voice can ignore it; nothing happens until `send_voice_packet_to_peer` is called. |

## The three API tiers

### 1. High-level (what game code calls)

```gdscript
# Once per network tick — usually from the listen-server's tick loop
GoolMultiplayerBridge.advance_tick(simulation_tick, server_time_ms)

# Firing a predicted gunshot — local plays immediately, peers get it via the active transport
var prediction_id := GoolMultiplayerBridge.fire_predicted_event(
        "gunshot_pistol", muzzle_position, 200)
# Cache prediction_id if you might need to cancel later

# If the server later says "miss" — cancel the prediction
Gool.cancel_predicted_event(prediction_id)

# Networked moving emitter (e.g. a teammate's footsteps emitter)
GoolMultiplayerBridge.update_replicated_transform_networked(
        emitter_handle, position, forward, velocity)

# Voice chat send (no-op on Steam if route_voice_packets = false)
GoolMultiplayerBridge.send_voice_packet_to_peer(
        peer_id, packet_bytes, seq_num, send_ts_ms)

# Host migration capture (called by host once per sync interval)
var snap := GoolMultiplayerBridge.capture_network_snapshot(
        PackedStringArray(["Music", "Sfx"]))
# ... ship snap over the wire ...

# Host migration apply (called by new host after migration)
GoolMultiplayerBridge.apply_network_snapshot(received_snap)
```

### 2. Low-level hooks (for custom transports)

```gdscript
# Switch to CUSTOM mode
GoolMultiplayerBridge.transport_mode = \
        GoolMultiplayerBridge.TransportMode.CUSTOM

# Listen for outbound packets and route through your own transport
GoolMultiplayerBridge.send_replicated_event.connect(
        func(payload):
            my_transport.send_event_to_all_peers(payload))

GoolMultiplayerBridge.send_voice_packet.connect(
        func(payload):
            steam_voice_api.send(payload.peer_id, payload.bytes))

# When YOUR transport receives an inbound packet, call the bridge's
# receive_* method — this dispatches to Gool with proper staleness check
func _on_my_transport_event_packet(data):
    GoolMultiplayerBridge.receive_replicated_event(
            data.sound_name, data.position, data.simulation_tick,
            data.priority, data.ts_ms)
```

### 3. Auto behaviors (each opt-out-able)

```gdscript
# In _ready of your main scene, or via the Autoloads inspector
GoolMultiplayerBridge.transport_mode = ...                # AUTO | MULTIPLAYER_API | CUSTOM | DISABLED
GoolMultiplayerBridge.auto_register_voice_sources = false  # take manual control
GoolMultiplayerBridge.route_voice_packets = false          # disable bridge voice routing (Steam)
GoolMultiplayerBridge.staleness_threshold_ms = 150         # tighter for competitive
```

## Practical recipes per transport

### Recipe: ENet (`ENetMultiplayerPeer`)

Defaults work. Bridge auto-detects the peer, routes everything via
`MultiplayerAPI` RPCs. Nothing to configure.

```gdscript
func host_match():
    var peer := ENetMultiplayerPeer.new()
    peer.create_server(7777, 4)
    multiplayer.multiplayer_peer = peer
    # Bridge automatically connects to peer_connected/disconnected.

func _on_player_joined(peer_id):
    print("Player ", peer_id, " joined — voice source registered")
```

### Recipe: Steam P2P (`SteamMultiplayerPeer` from a community asset)

`MultiplayerAPI` works the same. Disable bridge voice routing so
Steam handles voice natively.

```gdscript
func _ready():
    GoolMultiplayerBridge.route_voice_packets = false  # Steam handles voice

# When Steam delivers a voice packet via Steamworks callback:
func _on_steam_voice_received(steam_id, bytes):
    Gool.submit_voice_packet(steam_id, bytes, sequence_num,
            send_ts_ms, Time.get_ticks_msec())
```

### Recipe: Custom protocol (your own UDP / relay-based / hybrid)

```gdscript
func _ready():
    GoolMultiplayerBridge.transport_mode = \
            GoolMultiplayerBridge.TransportMode.CUSTOM
    GoolMultiplayerBridge.send_replicated_event.connect(_send_event_via_my_protocol)
    GoolMultiplayerBridge.send_voice_packet.connect(_send_voice_via_my_protocol)

func _send_event_via_my_protocol(payload: Dictionary):
    var bytes := PackedByteArray()
    # ... serialize payload your way ...
    my_relay.broadcast(bytes)

func _on_my_protocol_event_packet(payload: Dictionary):
    GoolMultiplayerBridge.receive_replicated_event(
            payload.sound_name, payload.position,
            payload.simulation_tick, payload.priority, payload.ts_ms)
```

## What the bridge does NOT do

These are deliberately out of scope so the bridge stays small and
swappable:

- **Doesn't decide which peer is host.** Host migration selection
  is your transport layer's job. Bridge only handles state capture
  / apply once your code decides a migration is happening.
- **Doesn't validate replicated data.** No anti-cheat. Trusts every
  payload — your transport must reject malformed input before
  calling `receive_*`.
- **Doesn't manage peer lists, lobbies, matchmaking.** Surface
  `multiplayer.peer_connected/disconnected` if using `MultiplayerAPI`;
  manage your own lobby if using a custom transport.
- **Doesn't reach into MusicStateController for snapshots.** The
  music state field on `GoolNetworkSnapshot` is populated by game
  code, not auto-detected. Bridge doesn't know which prefab is
  managing music.
- **Doesn't throttle replication rate.** `update_replicated_transform_networked`
  is called as often as the caller calls it. Per the arch doc's
  "release 2 or 3 decision" on emitter update rate, that knob lives
  in the caller, not the bridge.

## Relationship to the existing `Gool.play_networked()`

`Gool.play_networked(sound_name, position)` shipped in earlier
gool versions and is kept as the "one-call simple SFX" path —
no prediction, no priority handling, just "play this on all peers."
It uses `MultiplayerAPI` directly under the hood (same RPC pattern
the bridge uses).

The bridge's `fire_predicted_event` is the "predicted event with a
returned prediction_id you can cancel later" path — the right
choice when:

- The firing client should hear it at 0ms (gunshots, footsteps,
  player abilities)
- The server might invalidate the prediction (failed shot, missed
  ability)
- You need to set custom priority for engine voice eviction

For ambient world sounds, music triggers, and non-predicted
networked SFX, `Gool.play_networked` is simpler and still fine.

Both paths preserve the local-first invariant.

## Open items for v0.47.0+

  binding extension once the sub-tick stamp system lands)
- **Active emitter enumeration** for full host-migration state
  rehydration (needs new C++ binding)

  already exists; bridge could auto-report per-peer)
